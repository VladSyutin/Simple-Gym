# AGENTS.md

## Назначение

Этот файл задает проектные правила для агента в `SimpleGym`.
Проект находится на раннем bootstrap-этапе, но базовый iOS scaffold уже создан: в репозитории есть `project.yml`, `SimpleGym.xcodeproj`, `SimpleGym/App`, `SimpleGym/DesignSystem` и стартовый SwiftUI foundation.

## Контекст проекта

- `SimpleGym` - iOS-приложение для трекинга силовых тренировок.
- Источник визуального намерения - Figma.
- Приоритет UX - нативность iOS, системное поведение и аккуратное использование визуального языка в духе Liquid Glass.
- Figma направляет композицию и polish, но не должна заставлять проект копировать каждый слой макета кастомным кодом.

## Базовые технологические правила

- По умолчанию использовать `SwiftUI` как основной UI-слой.
- Минимальный `deployment target` проекта зафиксирован на `iOS 26.0`, потому что именно с `iOS 26` Apple официально ввела Liquid Glass и связанные API для современного дизайна.
- `UIKit` подключать только когда:
  - нужный системный компонент или API недоступен в SwiftUI;
  - интеграция через UIKit дает заметно более нативный результат;
  - этого требует реальный системный сценарий iOS.
- Не добавлять сторонние UI-библиотеки, если системные фреймворки iOS уже покрывают задачу.
- Не принимать на веру наличие конкретного iOS API. Перед реализацией проверять deployment target, доступность API и реальные ограничения платформы.

## Стартовая структура проекта

Если структура еще не существует, агент должен создавать и поддерживать ее в таком виде:

- `SimpleGym/App/`
  Точка входа приложения, `App`, маршрутизация верхнего уровня, app-wide environment.
- `SimpleGym/Features/`
  Фичи по доменам, например `WorkoutLog`, `History`, `Analytics`, `Settings`.
- `SimpleGym/DesignSystem/Components/`
  Переиспользуемые UI-компоненты и композиционные primitives.
- `SimpleGym/DesignSystem/Tokens/`
  Цвета, типографика, отступы, радиусы, материалы, тени, motion-константы.
- `SimpleGym/DesignSystem/Styles/`
  View modifiers, button styles, container styles, glass/material wrappers.
- `SimpleGym/Core/`
  Общие модели, утилиты, сервисные абстракции, storage, date/format helpers.
- `SimpleGym/Resources/`
  Нефигурирующие в asset catalog исходники, если они действительно нужны.
- `SimpleGym/Assets.xcassets/`
  Цветовые наборы, изображения, иллюстрации и подготовленные ассеты для приложения.

## Правила организации UI

- Сначала искать системный iOS-паттерн, потом искать переиспользуемый локальный компонент, и только затем создавать новый кастомный UI.
- Для экранов и flows использовать нативные контейнеры вроде `NavigationStack`, `TabView`, `List`, `ScrollView`, `sheet`, `popover`, `toolbar`, если они подходят задаче.
- Feature-specific UI размещать внутри соответствующей папки в `SimpleGym/Features/`.
- Truly reusable UI размещать только в `SimpleGym/DesignSystem/Components/`.
- Не складывать feature logic в design system.
- Имена Swift-файлов и типов должны быть в `PascalCase`.
- Один основной компонент или экран - один файл, если нет явной причины держать иначе.
- Если создается новый reusable component, рядом должны появиться preview-сценарии для основных состояний.

## Правила токенов и стилизации

- IMPORTANT: Не хардкодить `hex`, `UIColor(...)`, `Color(red:green:blue:)`, размеры, радиусы и spacing прямо во feature-view, если значение повторяемое или семантическое.
- Все устойчивые визуальные значения выносить в `SimpleGym/DesignSystem/Tokens/`.
- Предпочитать семантические токены:
  - `ColorTokens`
  - `Spacing`
  - `Radius`
  - `Typography`
  - `ShadowTokens`
  - `MaterialTokens`
- Токены из Figma переводить в семантический слой приложения, а не переносить как набор сырых чисел.
- Если в макете используется glass-эффект, сначала искать решение через системные `Material`, blur, vibrancy и container styling, а не рисовать собственный "glassmorphism" с нуля.
- Liquid Glass-подобные эффекты должны улучшать иерархию интерфейса, а не существовать как чистая декорация.
- Не наслаивать тяжелые blur/material-эффекты без проверки читаемости, производительности и контраста.

## Типографика и layout

- Использовать системную типографическую логику iOS и масштабирование, совместимое с Dynamic Type.
- Не фиксировать размеры текста, если это ломает адаптивность и доступность.
- Поддерживать комфортные tap targets и safe-area-aware layouts.
- Предпочитать адаптивные layout primitives вроде `frame(maxWidth:)`, `containerRelativeFrame`, `safeAreaPadding`, `contentMargins`, а не жесткую верстку под один размер экрана.
- Не переносить в код Figma-координаты как абсолютное позиционирование, если это можно выразить через нормальный SwiftUI layout.

## Accessibility и системное поведение

- Интерактивные элементы должны иметь понятные accessibility labels, если они неочевидны из системного контекста.
- Проверять контраст, особенно на полупрозрачных glass/material поверхностях.
- Учитывать Dynamic Type, VoiceOver, Reduce Motion и Reduce Transparency, когда визуальный эффект может повлиять на usability.
- Не жертвовать доступностью ради буквального совпадения с Figma.

## Asset Handling

- IMPORTANT: Если Figma MCP возвращает localhost asset URL, использовать его как источник истины и не заменять плейсхолдерами.
- IMPORTANT: Не добавлять новые icon packages по умолчанию.
- Сначала проверять, можно ли использовать `SF Symbols`.
- Если иконка или иллюстрация приходит из Figma и не заменяется системным ассетом, сохранять ее в `SimpleGym/Assets.xcassets` или в согласованный ресурсный каталог.
- Предпочитать векторные или system assets растровым, когда это не ухудшает визуальную точность.

## Figma MCP Integration Rules

Эти правила обязательны для любой задачи, где экран, компонент или визуальный паттерн приходят из Figma.

### Required Flow

1. Сначала запускать `get_design_context` для конкретного node или screen.
2. Если контекст слишком большой или неясна структура, запускать `get_metadata`, затем повторно забирать только нужные узлы через `get_design_context`.
3. Запрашивать `get_screenshot` для точной визуальной сверки.
4. Только после этого начинать реализацию и загрузку нужных ассетов.
5. Переводить Figma output в `SwiftUI`/`UIKit` и локальные проектные conventions, а не копировать референсный код буквально.
6. Перед завершением работы сверять итоговый UI с Figma screenshot и фиксировать заметные осознанные расхождения.

### Figma-to-iOS Translation Rules

- Рассматривать Figma как описание visual intent, spacing rhythm, hierarchy и state design.
- Не переносить Tailwind/React-style code из MCP буквально; преобразовывать его в нативные iOS abstractions.
- Если Figma показывает компонент, похожий на системный `Tab Bar`, `Navigation Bar`, segmented control, sheet, card list или form pattern, реализовывать его через нативную iOS idiom.
- Если Figma-решение конфликтует с системным поведением iOS, accessibility или performance, выбирать ближайшую нативную адаптацию.
- Любое заметное отступление от макета нужно кратко объяснять в итоговом отчете и, если решение стало устойчивым, фиксировать в project memory.

## Правила создания компонентов

- Перед созданием нового reusable UI проверить, нельзя ли собрать экран из:
  - системных компонентов iOS;
  - уже существующих design system primitives;
  - локальной feature composition без выделения нового shared component.
- Новый design system component создавать только если он:
  - используется повторно;
  - закрывает устойчивый визуальный паттерн;
  - уменьшает дублирование или упрощает перенос экранов из Figma.
- Компоненты design system должны быть максимально семантическими и не содержать бизнес-логики тренировки.
- Компоненты, стили и токены должны поддерживать light/dark behavior, если продукт не зафиксирует иное.

## Состояние, данные и архитектура

- Не смешивать сетевую, storage или domain logic с layout-кодом экранов.
- View composition должна оставаться тонкой: бизнес-правила и преобразования данных выносить в отдельные domain/view-model слои, когда проект их введет.
- До появления согласованной архитектуры избегать чрезмерно сложных паттернов и premature abstraction.
- Если новая задача требует архитектурного решения, сначала предложить минимальную структуру, совместимую с текущими правилами.

## Performance Rules

- Избегать тяжёлых стеков из blur, overlays и масок, если можно получить похожий нативный результат проще.
- Не добавлять дорогие эффекты во вложенные scroll-heavy экраны без необходимости.
- Кэшировать и оптимизировать ассеты по мере появления реальных bottlenecks, а не заранее.

## Правила проверки результата

- Каждая заметная Figma-driven реализация должна сопровождаться:
  - визуальной сверкой с макетом;
  - проверкой нативности взаимодействия;
  - проверкой доступности базовых сценариев;
  - проверкой, что токены и reusable patterns не захардкожены в feature code.
- Если проектная структура или правила изменились, обновлять этот файл вместе с реализацией.

## Приоритет решений

Когда правила конфликтуют, применять такой порядок:

1. Системное поведение iOS и доступность.
2. Реальные ограничения API, производительности и deployment target.
3. Семантика продукта и UX-сценария.
4. Переиспользование локальных компонентов и токенов.
5. Визуальная близость к Figma на уровне конкретного слоя.
